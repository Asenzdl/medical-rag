## 1. 项目概述

### 1.1 项目背景

医疗问答系统，使用 BERT（ERNIE）模型进行意图识别（科室分类），结合 ES 检索和 Redis 缓存，提供高质量的医疗问答服务。

### 1.2 核心需求

| 需求 | 说明 |
|------|------|
| **意图识别** | 用户问题 → 科室分类 |
| **语义检索** | 根据问题检索相关答案 |
| **缓存优化** | 提高响应速度，降低资源消耗 |
| **高可用** | 支持降级处理 |

### 1.3 用户场景

```
用户问题: "高血压怎么治疗"
    │
    ├─→ 意图识别: 内科
    ├─→ 语义检索: 高血压相关问答
    └─→ 返回结果: 治疗方案、注意事项等
```

│  │    ├─ 1. Redis 查询 (缓存结果)                           │   │

│  │    ├─ 3. ES 检索 (带科室过滤)                            │   │
│  │    └─ 4. 返回结果                                        │   │
│  └─────────────────────────────────────────────────────────┘   │

│  │             │    │             │    │             │        │
│  │ 向量搜索    │    │ TensorRT    │    │ BM25 搜索   │        │
│  │ 语义匹配    │    │ 3ms/请求    │    │ 科室过滤    │        │

│                                                                 │
└─────────────────────────────────────────────────────────────────┘

### 2.2 分层职责

| 层级 | 职责 | 技术 |
|------|------|------|
| **API 层** | 接收请求、返回响应、认证授权 | FastAPI |
| **Service 层** | 业务逻辑、缓存策略、错误处理 | Python |
| **数据层** | 数据存储、数据检索、数据缓存 | Redis, ES, BERT |

### 2.3 调用链路

```
用户问题 → FastAPI → MedicalQAService → BERT → embedding → Redis → ES → 返回结果
    │
    ├─ BERT 意图识别 → category (~3ms)
    ├─ embedding 模型 → 向量 (~10ms)
    ├─ Redis 混合查询 → 命中 → 返回 (<1ms)
    └─ Redis 未命中 → ES 检索 → 缓存结果 → 返回 (10-50ms)
```

#### 文件位置

```
src/bert/
├── model/              # HuggingFace 模型文件夹
├── onnx/               # ONNX 导出目录
├── model.trt           # TensorRT Engine

├── build_engine.py     # TRT Engine 构建脚本
└── trt_engine.py       # TensorRT 推理引擎
```

#### 核心功能

| 方法 | 用途 | 性能 |
|------|------|------|
| `predict(text)` | 单条推理 | ~3ms |
| `predict_batch(texts)` | 批量推理 | ~2.4ms/条 |

#### 优化点

| 优化 | 说明 | 效果 |
|------|------|------|
| **Buffer 预分配** | 启动时分配 GPU buffer | 减少内存分配开销 |
| **Stream 复用** | 所有推理共用 CUDA stream | 减少 stream 创建开销 |
| **统一推理逻辑** | predict() 调用 predict_batch() | 代码减少 40% |

#### 性能对比

| 指标 | 优化前 | 优化后 | 提升 |
|------|--------|--------|------|
| 单条延迟 | ~12ms | ~3ms | **75% ↓** |
| 批量吞吐 | 85 qps | 470 条/秒 | **5.5x ↑** |

## 5. QA Service 设计

### 5.1 核心流程

```
用户问题: "高血压怎么治疗"
    │
    ▼
┌─────────────────────────────────────────────────────────────────┐
│  Step 1: BERT 意图识别 (~3ms)                                   │
│                                                                 │
│  输入: "高血压怎么治疗"                                         │
│  输出: {"label": "内科", "confidence": 0.9979}                  │
└─────────────────────────────────────────────────────────────────┘
    │
    ▼
┌─────────────────────────────────────────────────────────────────┐
│  Step 2: embedding 模型 (~10ms)                                 │
│                                                                 │
│  输入: "高血压怎么治疗"                                         │
│  输出: [0.1, 0.2, ..., 0.7]                                    │
└─────────────────────────────────────────────────────────────────┘
    │
    ▼
┌─────────────────────────────────────────────────────────────────┐
│  Step 3: Redis 混合查询 (<1ms)                                  │
│                                                                 │
│  @category:{内科}=>[KNN 5 @embedding $vec AS score]             │
│  命中 → 返回                                                    │
└─────────────────────────────────────────────────────────────────┘
    │
    ▼ 未命中
┌─────────────────────────────────────────────────────────────────┐
│  Step 4: ES 检索 (10-50ms)                                      │
│                                                                 │
│  query: "高血压怎么治疗"                                        │
│  filter: category = "内科"                                      │
│  top_k: 5                                                       │
└─────────────────────────────────────────────────────────────────┘
    │
    ▼
┌─────────────────────────────────────────────────────────────────┐
│  Step 5: 缓存结果 → Redis                                       │
└─────────────────────────────────────────────────────────────────┘
    │
    ▼
┌─────────────────────────────────────────────────────────────────┐
│  Step 6: 返回结果                                               │
└─────────────────────────────────────────────────────────────────┘
```

### 5.2 耦合度分析

#### 方案对比

| 方案 | 特点 | 优点 | 缺点 |
|------|------|------|------|
| **A. 简单封装** | 直接依赖具体类 | 简单、快速 | 耦合度高 |
| **B. 依赖注入** | 通过参数传入 | 松耦合、可测试 | 代码稍多 |
| **C. Protocol 接口** | 定义抽象接口 | 最灵活、可替换 | 代码最多 |

#### 选择：方案 B（依赖注入）

**理由**：
1. 比简单封装更灵活
2. 比 Protocol 接口更简单
3. 支持单元测试
4. 支持组件替换

#### 实现方式

```python
class MedicalQAService:
    """医疗问答服务 - 依赖注入"""
    
    def __init__(self, bert_engine, es_store, cache):
        self.bert = bert_engine
        self.es = es_store
        self.cache = cache
    
    async def answer(self, question: str) -> dict:
        # 1. 查缓存
        # 2. BERT 意图识别
        # 3. ES 检索
        # 4. 缓存结果
        # 5. 返回结果
```

### 5.3 依赖注入 vs 紧耦合

#### 紧耦合

```python
# ❌ 紧耦合：想换成 CPU 推理，必须改代码
class MedicalQAService:
    def __init__(self):
        self.bert = TRTInferenceEngine()  # 写死了
        # 想换成 OptimumEngine？改代码！
```

#### 依赖注入

```python
# ✅ 依赖注入：想换成 CPU 推理，只需传入不同实现
class MedicalQAService:
    def __init__(self, bert_engine, es_store, cache):
        self.bert = bert_engine
        # 想换成 OptimumEngine？传入 OptimumEngine() 即可！
```

## 6. 缓存策略

### 6.1 缓存粒度选择

| 粒度 | Key 设计 | Value | 优点 | 缺点 |
|------|----------|-------|------|------|
| **A. 意图缓存** | `intent:{question}` | `{label, confidence}` | 小、快 | 还要查 ES |
| **B. 结果缓存** | `qa:{question}` | `{intent, results}` | 一步到位 | 大、占内存 |
| **C. 两级缓存** | 两者都有 | - | 灵活 | 复杂 |

### 6.2 选择：缓存最终结果

**理由**：
1. 医疗知识相对稳定，可以设置较长 TTL
2. 热门问题命中率高，减少 ES 查询
3. 用户体验更好（缓存命中 < 1ms）

### 6.3 缓存 Key 设计

#### 问题本质

```
用户可能用不同方式问同一个问题:

    "高血压怎么治疗"
    "高血压如何治疗"        ← 同一个问题，不同表述
    "怎么治疗高血压"
    "高血压  怎么治疗"      ← 多余空格
    "高血压怎么治疗？"      ← 标点符号

精确匹配无法解决这个问题！
```

#### 解决方案

| 方案 | 解决程度 | 复杂度 | 说明 |
|------|----------|--------|------|
| **精确匹配** | ❌ 差 | 低 | 无法解决 |
| **归一化** | ⚠️ 一般 | 低 | 解决格式差异 |
| **向量搜索** | ✅ 好 | 中 | 解决语义差异 |

#### 最终方案：Redis 向量搜索

```
流程:
    │
    ├─ 1. 用户问题 → BERT 意图识别 → category
    ├─ 2. 用户问题 → embedding → 向量
    ├─ 3. Redis 混合查询: @category:{内科}=>[KNN 5 @embedding $vec]
    ├─ 4. 命中 → 返回缓存结果
    └─ 5. 未命中 → ES 检索 → 缓存结果 → 返回
```

### 6.4 缓存 TTL 设计

| 选项 | TTL | 优点 | 缺点 |
|------|-----|------|------|
| **短期** | 1-6 小时 | 数据新鲜 | 命中率低 |
| **中期** | 6-24 小时 | 平衡 | - |
| **长期** | 24 小时+ | 命中率高 | 数据可能过时 |

**选择**: 6 小时（可配置）

### 6.5 置信度阈值设计

```
BERT 意图识别结果:
    │
    ├─ confidence >= 0.8: 高置信度，使用科室过滤
    ├─ 0.7 <= confidence < 0.8: 中置信度，可选过滤
    └─ confidence < 0.7: 低置信度，不过滤
```

### 6.6 错误处理设计

```
选项 A: 抛异常
    │
    ├─ 优点: 调用方知道出错了
    └─ 缺点: 需要调用方处理异常

选项 B: 返回空结果
    │
    ├─ 优点: 不会中断流程
    └─ 缺点: 调用方不知道出错了

选项 C: 降级处理
    │
    ├─ BERT 失败 → 不过滤，直接 ES 搜索
    ├─ ES 失败 → 返回空结果
    └─ 优点: 尽量返回结果
```

**选择**: 降级处理

## 7. 决策记录

### 7.1 决策总览

| 决策点 | 决定 | 理由 |
|--------|------|------|
| **耦合度** | 方案 B（依赖注入） | 灵活、可测试 |
| **缓存粒度** | 缓存最终结果 | 简单、高效 |
| **缓存 Key** | Redis 向量搜索 | 解决语义相似度 |
| **缓存 TTL** | 6 小时（可配置） | 平衡新鲜度和命中率 |
| **置信度阈值** | 0.7 | 平衡精度和召回 |
| **错误处理** | 降级处理 | 尽量返回结果 |

### 7.2 决策详情

#### 决策点 1: 耦合度

**问题**: 如何设计 QA Service 的依赖关系？

**选项**:
- A. 简单封装
- B. 依赖注入
- C. Protocol 接口

**决定**: 方案 B（依赖注入）

**理由**:
1. 比简单封装更灵活
2. 比 Protocol 接口更简单
3. 支持单元测试
4. 支持组件替换

#### 决策点 2: 缓存粒度

**问题**: 缓存意图还是缓存最终结果？

**选项**:
- A. 意图缓存
- B. 结果缓存
- C. 两级缓存

**决定**: 缓存最终结果

**理由**:
1. 医疗知识相对稳定
2. 热门问题命中率高
3. 用户体验更好

#### 决策点 3: 缓存 Key

**问题**: 如何解决"问题问法不同"的问题？

**选项**:
- A. 精确匹配
- B. 归一化
- C. 向量搜索

**决定**: Redis 向量搜索

**理由**:
1. 解决语义相似度问题
2. 部署简单，性能好
3. 数据量小，够用
